\section{Simulation}

This section presents a detailed overview of the verification test cases designed based on the validation checklist. These simulation runs aim to confirm the operational correctness and robustness of the complete SPI communication system integrated with the autonomous CRC5 error detection mechanism.

\subsection{Slave Module Verification (Slave Module Check)}

The Slave module verification is derived from the validation checklist and the scenarios implemented in \texttt{tb\_SPI\_Slave\_CRC.v}. This part checks the key Slave-side functions, including correct 13-bit frame reception from the Master, CRC error detection, data transmission from the Slave back to the Master through \texttt{MISO}, the expected \texttt{ready}/\texttt{valid} behavior, and the active-low chip-select condition controlled by \texttt{n\_cs}.

\begin{itemize}
    \item Receiving a valid 13-bit frame from the Master, consisting of 8 data bits and a correct 5-bit CRC.
    \item Receiving an invalid 13-bit frame with an incorrect CRC.
    \item Transmitting data from the Slave back to the Master and verifying the transmitted data and CRC.
    \item The Slave's behavior when \texttt{n\_cs} is not asserted (active high) while the Master is transmitting.
\end{itemize}

\paragraph{Slave\_TH1 -- Valid data reception from Master}
In the first scenario, the Master sends a valid 13-bit frame to the Slave through \texttt{MOSI}. The transmitted payload is \texttt{0x3C} and the accompanying CRC field is \texttt{10010}, which matches the expected CRC5 result for this frame. The waveform confirms that the Slave reconstructs the 8-bit output correctly after the full transaction, satisfying the checklist item \texttt{slave\_rx\_correct\_crc}. At the end of the frame, \texttt{Slave\_valid} remains asserted, indicating that the received data passes CRC checking. The \texttt{Slave\_ready} signal also follows the expected behavior: it is active when the Slave is idle and becomes inactive while the frame is being received.

    \caption{The Master sends a valid 13-bit frame with data 0x3C and correct CRC, and the Slave accepts the frame successfully.}
\end{figure}

\paragraph{Slave\_TH2 -- Invalid data reception from Master}
The second scenario verifies the CRC error-detection path on the Slave side. The Master again sends a 13-bit frame with payload \texttt{0x3C}, but this time the CRC field is intentionally corrupted to \texttt{10011}. The Slave still samples all incoming bits on \texttt{MOSI} and reconstructs the frame timing correctly, but at the final validation cycle \texttt{Slave\_valid} deasserts to \texttt{0}. This behavior matches the checklist item \texttt{slave\_rx\_wrong\_crc} and confirms that the Slave rejects frames whose 5-bit CRC field is incorrect.

\end{figure}

\paragraph{Slave\_TH3 -- Data transmission from Slave to Master}
The third scenario verifies the transmit behavior of the Slave, corresponding to the checklist item \texttt{slave\_tx\_data\_to\_master}. In this case, the Slave loads \texttt{0xA5} into its internal transmit buffer and then shifts the data back to the Master on \texttt{MISO} while \texttt{n\_cs} remains active low. The waveform shows that the Master-side receiver observes the correct 8-bit payload, and the full 13-bit transmitted frame contains the expected CRC field generated by the Slave. This confirms that the Slave correctly serializes its buffered data and appends the CRC bits during transmission.

    \caption{The Slave transmits data 0xA5 to the Master through \texttt{MISO} together with the correct CRC bits.}
\end{figure}

\paragraph{Slave\_TH4 -- Slave not selected while Master transmits}
The fourth scenario verifies the active-low chip-select behavior and complements the checklist items \texttt{slave\_ready\_behavior} and \texttt{slave\_chip\_select\_behavior}. Here, the Master still drives a 13-bit frame on \texttt{MOSI}, but the Slave input \texttt{n\_cs} is kept high for the entire transaction. Because the Slave is not selected, it does not shift in the incoming bits, does not accept the frame, and remains inactive from the communication point of view. The waveform confirms that the Slave only participates in reception or transmission when \texttt{n\_cs} is asserted low, while deselection keeps the internal receive operation disabled and preserves the expected idle behavior.

\subsection{Master Module Verification (Master Module Check)}

The Master module verification is derived from the validation checklist and the implemented scenarios in \texttt{tb\_SPI\_Master\_CRC.v}. This part checks the key Master-side functions, including correct 13-bit frame reception from a virtual Slave, CRC error detection, active-low Slave selection, control sequencing through \texttt{CNTL}, transfer-clock generation on \texttt{Master\_SCLK}, and the expected \texttt{READY}/\texttt{VALID} behavior before, during, and after transmission.

\begin{itemize}
    \item Receiving a valid 13-bit frame from a virtual Slave, including 8 data bits and a correct 5-bit CRC.
    \item Receiving a 13-bit frame with an incorrect CRC and verifying that the Master detects the error correctly.
    \item Selecting the target Slave through active-low \texttt{Master\_SS}, including the invalid-selection case where no Slave is chosen.
    \item Loading transmit data into the Master and sending the data together with CRC bits through \texttt{MOSI}.
    \item Verifying the control flow of \texttt{CNTL}, the generation of \texttt{Master\_SCLK}, and the expected \texttt{READY}/\texttt{VALID} behavior across the transfer.
\end{itemize}

\paragraph{Master\_TH1 -- Valid data reception from Slave}
In the first scenario, the Master verifies a correct receive transaction from the Slave side. The testbench selects Slave index 0 by applying \texttt{INPUT = 0x00} with \texttt{CNTL = 2}, then a virtual Slave transmits a 13-bit frame on \texttt{MISO} with data \texttt{0x3C} and CRC \texttt{10010}. This is the same valid 13-bit pattern used in the first Slave-side verification case, allowing direct comparison between the receive behavior of both modules. During \texttt{CNTL = 3}, the Master samples the incoming frame using \texttt{Master\_SCLK}; at the final checking cycle, \texttt{Master\_output} matches the expected 8-bit payload and \texttt{Master\_valid} remains high, confirming successful CRC verification. The waveform also shows that \texttt{Master\_ready} is asserted while the Master is idle or configuring the transaction and deasserted during the active transfer interval, which is consistent with the design and the checklist expectations.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{\detokenize{images/Master_check/Master_TH1.png}}
    \caption{The Master receives a valid 13-bit frame from the virtual Slave (data = 0x3C, correct CRC), and \texttt{Master\_valid} stays high at the final check cycle.}
\end{figure}

\paragraph{Master\_TH2 -- Invalid CRC reception and invalid Slave selection}
The second scenario checks the error-detection path of the Master. The virtual Slave sends another 13-bit frame to the Master, but the CRC field is intentionally incorrect. In this testbench, the transmitted data is \texttt{0xCC} and the CRC pattern is corrupted, so the CRC5 checker inside the Master rejects the frame. As a result, at the last checking clock of the transfer, \texttt{Master\_valid} falls to \texttt{0}, indicating CRC failure exactly as required by the checklist item \texttt{master\_rx\_wrong\_crc}. At the same time, the Slave-select logic is also verified using an invalid selection input: when \texttt{CNTL = 2} and \texttt{INPUT = 0x0F}, the selected index is outside the valid range for 8 Slaves, so \texttt{Master\_SS} remains \texttt{0xFF}. This means no Slave is selected, which correctly matches the active-low Slave-select specification.

\end{figure}

\paragraph{Master\_TH3 -- Master transmits data to Slave 3}
The third scenario verifies the transmit path from the Master to a selected Slave. First, the testbench loads \texttt{0x33} into the Master transmit buffer using \texttt{CNTL = 1}. Next, Slave index 3 is selected with \texttt{CNTL = 2} and \texttt{INPUT = 0x03}. Based on the active-low select logic implemented in \texttt{SPI\_Master\_CRC}, this produces \texttt{Master\_SS = 11110111}, meaning only Slave 3 is enabled. The transfer then starts when \texttt{CNTL = 3}, and the outgoing 13-bit frame on \texttt{MOSI} is captured by the testbench. The waveform confirms that the Master transmits the 8-bit payload \texttt{0x33} together with its CRC bits in the expected order, while \texttt{Master\_SCLK} is generated only during the transfer phase. Therefore, this case validates the checklist items related to \texttt{master\_tx\_data\_to\_slave}, \texttt{master\_slave\_select}, \texttt{master\_cntl\_state\_control}, and \texttt{master\_sclk\_generation}.

\subsection{Integrated System Verification (Full SPI Communication Check)}

\noindent The integrated system verification uses the complete Master-Slave connection in \texttt{tb\_Com\_Full\_Module.v} to validate end-to-end SPI behavior. This subsection covers the main checklist items for the full communication system: \texttt{master\_to\_slave\_transfer}, \texttt{slave\_to\_master\_transfer}, \texttt{full\_duplex\_transfer}, \texttt{crc\_error\_detection\_integrated}, \texttt{ss\_sclk\_sync\_integrated}, and \texttt{idle\_to\_transfer\_recovery}. The simulation is organized into three practical communication phases that together verify the complete integrated behavior of the design.

\begin{itemize}
    \item Transmitting data from the Master to the Slave and verifying correct reception on the Slave side.
    \item Transmitting data from the Slave to the Master and verifying correct reception on the Master side.
    \item Verifying simultaneous full-duplex communication where both modules exchange data within the same frame.
    \item Checking CRC validation behavior in the integrated communication path.
    \item Confirming synchronization between \texttt{SS}, \texttt{SCLK}, \texttt{MOSI}, and \texttt{MISO} during active transfer.
    \item Verifying that the system transitions cleanly from idle to transfer and returns to idle without stale data or control errors.
\end{itemize}

\paragraph{Full\_Module\_TH1 -- Master-to-Slave transfer}
In the first integrated scenario, the Master sends data to the Slave through the normal SPI path. The testbench loads \texttt{0xD4} into the Master transmit input, selects Slave 0 by driving \texttt{INPUT = 0x00} during the select phase, and then starts the transfer. The waveform confirms that the data placed on \texttt{MOSI} is correctly captured by the Slave, and the received Slave output matches the transmitted Master payload. At the end of the frame, the Slave-side CRC validation signal indicates successful reception, confirming the integrated \texttt{master\_to\_slave\_transfer} path. This phase also shows that the transaction occurs only when \texttt{SS[0]} is active low and \texttt{SCLK} is generated, which verifies the expected SS-SCLK synchronization behavior.

\paragraph{Full\_Module\_TH2 -- Slave-to-Master transfer}
The second integrated scenario verifies the reverse communication path from the Slave back to the Master. Here, the Slave loads \texttt{0xA5} into its transmit buffer, while the Master keeps the Slave selected and performs a receive transaction. The waveform shows that the Slave drives the data onto \texttt{MISO}, and the Master reconstructs the expected 8-bit output correctly after one full frame. This confirms the \texttt{slave\_to\_master\_transfer} behavior and demonstrates that the Master-side receive and CRC-checking logic operate correctly in the integrated connection. The ready/valid behavior also follows the expected protocol: the Master leaves the idle state, completes the transfer, and then returns to ready state once \texttt{CNTL} is deasserted back to idle.

\paragraph{Full\_Module\_TH3 -- Full-duplex integrated transfer}
The final integrated scenario verifies simultaneous bidirectional communication. The Master loads \texttt{0x33} for transmission while the Slave loads \texttt{0xCC} into its own transmit buffer. When the transfer begins, the Master sends data to the Slave on \texttt{MOSI} while the Slave simultaneously sends data back to the Master on \texttt{MISO}. The resulting waveform confirms that both modules receive the opposite-side payload correctly after a single complete transaction, which verifies the \texttt{full\_duplex\_transfer} requirement. In addition, this phase demonstrates correct transition from idle to transfer and back to idle without stale data or counter errors, while \texttt{READY} and \texttt{VALID} remain consistent with the protocol specification. Together with the previous phases, this confirms that the integrated system satisfies the intended functional behavior of the complete SPI communication architecture.

\subsection{Loop Back Verification (Loop Back Check)}

The loopback verification focuses on continuous data circulation between the Master and Slave after both modules are connected together. Based on \texttt{tb\_SPI\_Circular\_Loopback.v}, this test checks the \texttt{loopback\_transfer} behavior in the checklist by confirming that the Master can first send data to the Slave, then the Slave stores the received value, reloads it into its transmit buffer, and returns the same value back to the Master in the next transaction. The waveform also confirms that the handshake signals operate according to the specification: \texttt{m\_ready} is asserted in the idle or setup phases, deasserted during transfer, and \texttt{m\_valid} indicates successful frame completion after CRC checking.

\begin{itemize}
    \item Sending data from the Master to the Slave in the first transaction of the loopback sequence.
    \item Storing the received value inside the Slave and reloading it into the Slave transmit path.
    \item Returning the same data from the Slave back to the Master in the next transaction.
    \item Verifying continuous circular communication behavior between both modules.
    \item Checking that \texttt{SS}, \texttt{SCLK}, \texttt{MOSI}, \texttt{MISO}, \texttt{ready}, and \texttt{valid} operate consistently throughout the loopback process.
\end{itemize}

\paragraph{LoopBack\_TH1 -- Circular loopback transfer}
In this testbench, the initial payload loaded into the Master is \texttt{0x99}. During the first loop, the Master loads the data, selects Slave 0, and transmits the frame to the Slave through \texttt{MOSI}. The Slave receives and stores the transmitted value correctly. In the second loop, the Slave loads its output data back into the transmit path and sends the same value \texttt{0x99} to the Master through \texttt{MISO}. At the same time, the Master also reloads its input path to continue the communication cycle. The final waveform confirms that the data returning to the Master matches the original transmitted value, proving that the circular ping-pong loopback mechanism works correctly. This case also demonstrates correct synchronization between \texttt{SS}, \texttt{SCLK}, \texttt{MOSI}, and \texttt{MISO}, since data movement only occurs while the Slave is selected and the transfer clock is active.
